# BUG-010: Express 스캐너 — auth 미들웨어 이름 정확일치만 인정 (camelCase 누락)

- **상태**: Fixed (Phase139)
- **등급**: 정확도 결함 (auth/security 오분류 — 전부 public)
- **현재 영향**: 실재 — Ghost 583건 전부 `public`(실제는 admin API 전체 인증 필요)
- **발견**: SaaS 추출 전수조사(Ghost) 중 (2026-05-30)
- **파일**: `internal/scanner/express/classify_auth_middleware.go`

## 증상

라우트의 인증 여부 판정이 **고정 소문자 이름 집합과 정확 일치**만 인정해, 실제 auth
미들웨어를 못 알아보고 모든 라우트를 `authLevel=public`으로 표기한다.

Ghost 실측:
- 미들웨어 `mw.authAdminApi`(소문자화 `authadminapi`)가 집합(`auth`,`authenticate`,…)과
  불일치 → 인증 필요한 admin API 전체가 `public`.
- middleware 필드에는 `mw.authAdminApi`가 raw로 잡혀 있으나 authLevel 산출에 미사용.

## 근본 원인 (코드)

`classify_auth_middleware.go` — auth로 간주할 미들웨어 이름을 고정 소문자 set으로 두고
**정확 일치**만 검사. `authAdminApi`/`requireAuth`/`ensureLoggedIn` 등 camelCase·접두형
실무 명명을 포착 못 함. 이미 캡처된 middleware 이름을 활용하지도 않음.

## 권장 수정

- 이름 매칭을 **부분일치/접두어 기반**(`auth`로 시작하거나 포함, `require*`/`ensure*Auth`
  등)으로 확장.
- 이미 추출된 `middleware` 필드의 이름을 authLevel 산출에 반영.

## 비고

- NestJS는 `@UseGuards`로 비교적 정확히 인증을 잡는 편(Immich에서 242/242 정확)이라
  본 결함은 Express 스캐너 한정.
