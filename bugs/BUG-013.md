# BUG-013: Django 스캐너 — 커스텀 Base 클래스 상속체인 미해소 → 단일 GET 추락

- **상태**: Fixed (Phase142)
- **등급**: 정확도 결함 (ViewSet/APIView 인식 실패 → fallback 단일 GET)
- **현재 영향**: 실재 — Plane 333건 중 325건이 fallback 단일 GET
- **발견**: SaaS 추출 전수조사(Plane) 중 (2026-05-30)
- **파일**: `internal/scanner/django/is_viewset_subclass.go:6-26`, `internal/scanner/django/is_apiview_subclass.go:6-28`, `internal/scanner/django/build_plain_view_endpoints.go:14`

## 증상

뷰 클래스가 **DRF 베이스를 직접 상속하지 않고 프로젝트 커스텀 중간 베이스를 거치면**
ViewSet/APIView로 인식하지 못해, 클래스 본문의 `def get/post/put/patch/delete`를 읽지
못하고 **단일 GET** endpoint로 추락한다.

Plane 실측:
- 구조: `ProjectViewSet → BaseViewSet → ModelViewSet`,
  `XxxEndpoint → BaseAPIView → APIView` (`plane/app/views/base.py:48,149`).
- codist는 `BaseViewSet`/`BaseAPIView`를 몰라 `findViewSet`/`findAPIView`가 nil →
  325 라우트가 `buildPlainViewEndpoints`로 추락 → 전부 단일 GET.

## 근본 원인 (코드)

- `is_viewset_subclass.go:6-26` / `is_apiview_subclass.go:6-28` — **직접 부모 이름**만
  하드코딩 set과 대조. 전이적(transitive) 상속 해소 없음.
- `build_single_url_entry_endpoints.go:22` → `buildPlainViewEndpoints`로 추락.
- `build_plain_view_endpoints.go:14` — fallback이 **무조건 단일 GET** 발급.

## 권장 수정

- `isViewSetSubclass`/`isAPIViewSubclass`가 repo 내 정의된 중간 베이스를 따라 올라가
  DRF 루트(`ModelViewSet`/`ViewSet`/`APIView` 등)까지 **전이적으로 판정**.
- 해소 성공 시 클래스 본문의 HTTP 메서드(`def get/post/...`)를 endpoint로 전개.

## 비고

- BUG-012(as_view dict)와 함께 고치면 Plane method recall이 크게 회복.
- 커스텀 `Base*` 패턴은 대다수 실무 Django 프로젝트의 표준 → 영향 광범위.
- Django 스캐너는 response/auth 정보를 거의 방출하지 않음(별도 한계).
