# BUG-008: Actix 스캐너 — `fn() -> Scope` 함수 간접등록 빌더체인 미해석

- **상태**: Fixed (Phase137)
- **등급**: 정확도 결함 / 기능 공백 (false negative — 빌더 라우트 대량 누락)
- **현재 영향**: 실재 — AppFlowy-Cloud에서 185 빌더 라우트 중 1개만 추출
- **발견**: SaaS 추출 전수조사(AppFlowy) 중 (2026-05-30)
- **파일**: `internal/scanner/actix/process_service_arg.go:10-19`, `internal/scanner/actix/is_top_level_service_call.go:24-27`

## 증상

Actix 스캐너가 **attribute 매크로 방식**(`#[get("...")]`)은 정상 추출하나, 실무에서
흔한 **`pub fn xxx_scope() -> Scope { web::scope(...).service(web::resource(...).route(web::get().to(h))) }`
형태로 라우트를 모듈화하고 `App.service(xxx_scope())`로 등록**하는 빌더체인을 전혀
못 잡는다.

AppFlowy 실측:
- attribute 매크로: 코드 전체에 **1개**(`ws.rs:40 #[get]`) → codist가 잡은 유일한 endpoint.
- 빌더 라우트(`web::method().to()`): **185개**(get 80/post 65/put 19/delete 17/patch 4) → 전부 누락.
- 또한 잡힌 1개도 path가 `/{token}/{device_id}`로 **scope prefix `/ws`가 빠짐**.

## 근본 원인 (코드)

1. `process_service_arg.go:10-19` — `.service()` 인자의 `findCallRoot`가
   `web::scope`/`web::resource`일 때만 처리. AppFlowy의 top-level `.service()` 인자는
   `workspace_scope()` 같은 **일반 함수 호출**이라 `callRoot=""` → 미매치 → 스킵.
   `*_scope()` 함수 호출을 그 정의(반환 `web::scope(...)`)로 되짚는 기능이 없다.
2. `is_top_level_service_call.go:24-27` — `*_scope()` 함수 내부의
   `web::scope(...).service(A).service(B)` 체인은 receiver가 web::scope 체인이라
   `isTopLevelServiceCall`이 전부 false → 추출 진입점이 되지 못함.

설계 전제는 `App::new().service(web::scope(...).service(...))`처럼 `web::scope(...)`가
`.service()`의 **인라인 인자로 직접** 등장하는 것(AppFlowy엔 0건).

## 권장 수정

1. `process_service_arg.go`: `.service()` 인자가 일반 함수 호출이면 그 함수 정의를
   찾아(반환 `web::scope(...)` 체인) entry로 삼아 재귀.
2. `is_top_level_service_call.go`: `pub fn ...() -> Scope` 본문의 `web::scope(...).service(...)`
   체인을 빌더 추출 진입점으로 인정.
3. path: `*_scope()` 정의의 `web::scope` 첫 인자를 prefix로 전파(scope prefix 합성).
