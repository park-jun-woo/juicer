# BUG-001: `resolveCallTarget` nil-`info` 가드 누락 (echo 스캐너)

- **상태**: Fixed (Phase130)
- **수정 요약**: `resolve_call_target.go`의 `resolveCallTarget` 함수 첫 줄에 `if info == nil { return token.NoPos }` 가드 추가 (형제 함수와 일관성 확보).
- **등급**: 잠재적(latent) nil 포인터 역참조 / 방어코드 일관성 결함
- **현재 크래시 위험**: 없음 (현 코드 경로에서 발현 안 됨)
- **발견**: tsma 테스트 완주 작업 중 (2026-05-30)
- **파일**: `internal/scanner/echo/resolve_call_target.go:12`

## 증상

`resolveCallTarget(call *ast.CallExpr, info *types.Info)`가 `info == nil`
가드 없이 `info.Selections` / `info.Uses`를 역참조한다. `info`가 nil로
전달되면 nil 포인터 panic이 발생한다.

```go
func resolveCallTarget(call *ast.CallExpr, info *types.Info) token.Pos {
	switch fn := call.Fun.(type) {
	case *ast.SelectorExpr:
		if sel, ok := info.Selections[fn]; ok {   // ← info가 nil이면 panic
			return sel.Obj().Pos()
		}
		if use, ok := info.Uses[fn.Sel]; ok {
			return use.Pos()
		}
	case *ast.Ident:
		if use, ok := info.Uses[fn]; ok {          // ← 동일
			return use.Pos()
		}
	}
	return token.NoPos
}
```

## 근거 — 형제 함수와의 일관성 결함

같은 `internal/scanner/echo` 패키지의 다른 함수들은 모두 `info == nil`
가드를 갖고 있다. `resolveCallTarget`만 빠져 있다.

- `resolve_expr_const.go:13` — `if info == nil`
- `resolve_expr_type.go:12` — `if info == nil`
- `resolve_status_code.go:17` — `if info == nil`
- `resolve_response_type.go:12` — `if info == nil`
- `resolve_bind_type.go:13` — `if len(call.Args) == 0 || info == nil`
- `check_one_depth_call.go:12` — `if info == nil`
- `analyze_handlers.go:23` — `if info == nil`
- `echo_ctx_param_name_info.go:11` — `if ft.Params == nil || info == nil`
- `echo_router_param_at_index.go:18` — `if info == nil`

## 왜 현재는 안 터지나 (호출 경로 분석)

호출자는 두 곳뿐이다.

| 호출자 | `info` 출처 | nil 가능성 |
|--------|-------------|-----------|
| `check_one_depth_call.go:29` (`checkOneDepthCall`) | 함수 진입부 `if info == nil { return }` 가드 후 호출 | 불가능 (안전) |
| `rescan_callee_with_prefix_depth.go:16` (`rescanCalleeWithPrefixDepth`) | `ctx.info` = `pkg.TypesInfo` | 사실상 non-nil |

`ctx.info`는 `resolve_group_prefix_func.go:18`에서 `pkg.TypesInfo`로 채워진다.
`scan.go:21-27`의 `packages.Load` 모드에 `packages.NeedTypesInfo`가 켜져
있어, 이 플래그가 설정되면 로더가 패키지마다 `TypesInfo`를 항상 할당한다
(타입 에러가 있어도 non-nil). 따라서 현 코드에서 `ctx.info`가 nil이 되는
시나리오는 없다.

## 발현 조건 (회귀 위험)

- 누군가 `resolveCallTarget`을 nil `info`로 호출하는 새 호출자를 추가
- `scan.go`의 load 모드에서 `NeedTypesInfo`를 제거/변경
- 위 둘 중 하나라도 발생하면 즉시 panic 후보가 된다.
