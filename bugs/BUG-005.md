# BUG-005: NestJS — 중첩 DTO/enum 미추출 + 스키마 casing 불일치·배열 의사스키마

- **상태**: Fixed (Phase134)
- **등급**: 정확도/스펙충실도 결함 (스키마 대량 누락 + casing 불일치 + 배열 오표현)
- **현재 영향**: 실재 — Immich에서 스키마 ~202개 미추출
- **발견**: SaaS 추출 전수조사(Immich) 중 (2026-05-30)
- **파일 (두 레이어)**: ① 중첩 추출 = `internal/scanner/nestjs`(`resolve_all_dtos.go`/`extract_dto.go`),
  ② casing·`$ref`·배열표현 = **상위 `internal/scanner`**(`body_schema.go`/`response_schema.go`/`field_to_property.go`/`resolve_primitive_schema.go`/`lc_first.go`)

## 증상

1. **중첩 DTO/enum 미추출**: 최상위 request/response DTO는 잘 뽑지만, 그 속성 타입
   (중첩 object DTO·enum)을 재귀적으로 추출하지 않는다.
   - Immich: codist 스키마 194 vs 공식 359. 누락 202개 = enum 50 + 중첩 object DTO 152.
   - 누락 표본: `AssetOrder`, `AlbumUserRole`(enum), `AlbumUserResponseDto`,
     `AssetBulkUploadCheckItem`(중첩 DTO).
2. **$ref casing 불일치**: codist는 `assetBulkDeleteDto`(lowerCamel), 공식은
   `AssetBulkDeleteDto`(UpperCamel). exact-case 매칭 0개.
3. **배열 의사(pseudo)-스키마**: 배열을 `type:array + items:$ref` 대신 `xxxResponseDto[]`
   라는 이름의 스키마로 생성(codist에만 37개).

> **정정(검수 2026-05-31)**: "dangling $ref"는 현재 산출물에서 **재현되지 않는다**
> (실측 dangling 0 — emit가 의사스키마까지 자기참조적으로 정의해 `$ref` 대상이 전부 존재).
> 실제 결함은 ① 스키마 대량 미추출, ② casing 불일치, ③ 배열 의사스키마 셋이다.
> dangling은 잠재 위험이지 현 증상이 아님.

## 근본 원인 (코드 — 두 레이어)

- **(A) 중첩 미추출 (`internal/scanner/nestjs`)**: `extractDTO`가 extends/배럴은 따라가나
  속성 타입(중첩 DTO·enum)은 큐잉 안 함 → 최상위 DTO만 1-레벨 추출, 나머지는 인라인.
- **(B) casing (`internal/scanner`)**: `body_schema.go`/`response_schema.go`/`field_to_property.go`가
  스키마명/`$ref`를 `lcFirst()`(`lc_first.go`)로 lowerCamel 생성 → 공식 UpperCamel과 불일치.
- **(B) 배열 의사스키마 (`internal/scanner`)**: `resolve_primitive_schema.go`가 Go식 `[]` 접두만
  벗기고 TS식 `[]` 접미는 못 벗겨 `xxxResponseDto[]` 이름의 스키마가 등록됨(올바른
  `type:array+items:$ref` 아님). Immich에서 36개 확인.

## 권장 수정

1. DTO 추출 파이프라인에서 속성 타입(중첩 DTO)·enum을 **재귀적으로 등록**.
2. 스키마 이름/`$ref` casing 통일.
3. 배열은 `type:array + items:$ref` 로 생성(의사스키마 제거).

## 비고

- 라우트(operation) 정확도(BUG-003/004와 별개)는 매우 높으나, 스키마 레이어가 약점.
