# BUG-003: NestJS 스캐너 — `@Controller(Enum/멤버표현식)` 경로 미해석 (RouteKey)

- **상태**: Fixed (Phase132)
- **등급**: 정확도 결함 (경로 손상 → 다수 endpoint 매칭 실패)
- **현재 영향**: 실재 — Immich에서 37 operation 경로 손상
- **발견**: SaaS 추출 전수조사(Immich) 중 (2026-05-30)
- **파일**: `internal/scanner/nestjs/controller_prefix.go:8`, `internal/scanner/nestjs/parse_decorator_call.go:31`

## 증상

`@Controller(RouteKey.Asset)`처럼 컨트롤러 경로를 **enum/상수(멤버표현식)**로 주면
codist가 그 값을 해석하지 못하고 path를 문자열 그대로 `"/api/RouteKey.Asset"`으로
출력한다. 실제 값은 `/api/assets`.

Immich 실측:
- `RouteKey` 정의: `server/src/enum.ts:565` → `enum RouteKey { Asset='assets', User='users' }`
- `/api/RouteKey.Asset` (+하위 24 ops), `/api/RouteKey.User` (+하위 13 ops)
- 합계 **37 operation**이 경로 손상. 경로 정규화 전 recall 99.6% → **84.4%로 추락**.

## 근본 원인 (코드)

- `parse_decorator_call.go:31-32` — `@Controller()` 인자가 `member_expression`일 때
  `nodeText`로 원시 문자열을 그대로 저장하고 enum 해석을 하지 않음.
- `controller_prefix.go:8-16` — `controllerPrefix()`가 `d.arg`(="RouteKey.Asset")를
  그대로 prefix로 반환. 식별자/멤버표현식 분기 없음.

스캐너는 이미 DTO 필드용 enum 해석기를 보유한다
(`collect_enum_member_values.go`, `extract_enum_assignment_value.go`).

## 권장 수정

`controllerPrefix`(및 `parseDecoratorCall`)에서 인자가 `member_expression`/`identifier`이면
기존 enum 해석기(`collect_enum_member_values.go`)를 재사용해 실제 문자열로 해석.
→ Immich recall 84%→99.6% 회복.

## 비고

- 메서드 데코레이터(`@Get(RouteKey.X)`)에도 동일 문제 가능 — 함께 처리 권장.
